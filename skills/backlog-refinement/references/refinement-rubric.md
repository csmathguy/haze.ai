# Refinement Rubric

## Core Questions

Ask only the questions that affect the decision:

- What user or operational outcome is this item for?
- What would count as success?
- What is in scope and what is explicitly out of scope?
- What depends on this item being done first?
- What risk or ambiguity is blocking confidence?
- Is this item actually one item, or several?

## Decision Guide

- Keep: the item is useful and can be shaped into a valid backlog item.
- Clarify: the item has value, but it is too vague to execute safely.
- Split: the item contains more than one independently shippable outcome.
- Block: the item is valid, but an external dependency or missing decision prevents execution.
- Drop: the item has low value, is a duplicate, or is not worth the likely cost.

## Readiness Signals

- acceptance criteria describe observable outcomes
- dependencies are explicit
- the scope fits one active slice
- the item can be validated
- the backlog item is not hiding future work that should be separate

## Anti-Patterns

- asking implementation questions before the outcome is known
- polishing every backlog item to the same level
- using refinement as a gate that prevents movement
- leaving duplicates or follow-ups implicit
- treating a vague idea as ready because it sounds important
